# ДЗ 2: библиотека промпт-шаблонов

Все шаблоны используют структуру **RTCF**: Role, Task, Context, Format. Они рассчитаны на Codex и проект FocusTrack AI.

## Шаблон 1. Генерация React-компонента

```text
Role:
Ты senior React + TypeScript разработчик в проекте FocusTrack AI.

Task:
Создай компонент <COMPONENT_NAME> для <USER_SCENARIO>.

Context:
- Стек: React, TypeScript, Vite, Tailwind.
- Домен: цели, задачи, прогресс, weekly review.
- Используй существующие типы и стиль проекта.
- Нужны состояния loading, empty и error, если они применимы.
- Не добавляй новую библиотеку без отдельного обоснования.

Format:
- Сначала краткий план.
- Затем список файлов.
- Затем код.
- В конце проверки, которые нужно выполнить.
```

### Пример использования

```text
Создай GoalSummaryCard для отображения цели, процента выполнения, streak и ближайшей задачи.
```

## Шаблон 2. Рефакторинг без изменения поведения

```text
Role:
Ты senior TypeScript engineer, отвечающий за безопасный рефакторинг.

Task:
Отрефактори <FILE_OR_MODULE>, сохранив текущее поведение.

Context:
- Проект: FocusTrack AI.
- Не меняй публичные контракты без явного перечисления.
- Сначала найди дублирование, сложные условия и слабые имена.
- Не расширяй scope задачи.

Format:
- Кратко опиши текущую проблему.
- Предложи минимальный план.
- Внеси правки.
- Перечисли проверки и возможные риски.
```

### Пример использования

```text
Отрефактори calculateGoalProgress так, чтобы обработка пустого списка задач была явной.
```

## Шаблон 3. Тесты

```text
Role:
Ты frontend test engineer для React + TypeScript проекта.

Task:
Добавь тесты для <COMPONENT_OR_FUNCTION>.

Context:
- Покрой happy path, empty/error state и один edge case.
- Не тестируй внутреннюю реализацию, проверяй поведение.
- Если тестовый стек ещё не создан, предложи минимальный вариант и не внедряй его без подтверждения.

Format:
- Список сценариев.
- Файлы тестов.
- Команды проверки.
- Краткое объяснение, какой риск покрыт.
```

### Пример использования

```text
Добавь тесты для GoalCard: отображение прогресса, отсутствие задач, просроченная задача.
```

## Шаблон 4. Исправление ошибки

```text
Role:
Ты инженер, который диагностирует баги в fullstack-приложении.

Task:
Найди и исправь причину ошибки <BUG_DESCRIPTION>.

Context:
- Сначала воспроизведи или локализуй проблему.
- Не исправляй симптом, пока не сформулирована вероятная причина.
- Проверь связанные состояния UI и API.
- Секреты и ключи не выводи в логи и документы.

Format:
- Причина.
- Исправление.
- Изменённые файлы.
- Проверки.
- Остаточный риск.
```

### Пример использования

```text
Weekly Review показывает пустой отчёт, если за неделю есть выполненные задачи.
```

## Шаблон 5. Декомпозиция задачи

```text
Role:
Ты tech lead и product engineer FocusTrack AI.

Task:
Разбей задачу <FEATURE> на маленькие шаги реализации.

Context:
- MVP: SMART goal input, AI clarify, AI plan, dashboard, Weekly AI Review.
- Учитывай зависимости между frontend, Supabase и Edge Functions.
- Выдели must-have и nice-to-have.
- Не предлагай RAG как обязательную часть MVP без отдельного решения.

Format:
- Цель фичи.
- Список шагов в порядке реализации.
- Зависимости.
- Проверки готовности.
- Что можно отложить.
```

### Пример использования

```text
Разбей реализацию AI-декомпозиции цели в задачи на frontend, database и Edge Function.
```

## Шаблон 6. Supabase Edge Function для AI

```text
Role:
Ты backend engineer, который пишет Supabase Edge Functions для AI-вызовов.

Task:
Спроектируй Edge Function <FUNCTION_NAME>.

Context:
- OpenRouter вызывается только на серверной стороне.
- Вход и выход должны быть JSON.
- Учитывай auth пользователя и RLS.
- Логируй AI-сессию без секретов и приватных токенов.
- Prompt injection нужно ограничить системной инструкцией и строгим использованием входного JSON.

Format:
- Контракт входа.
- Контракт выхода.
- Ошибки и статусы.
- Псевдокод или код функции.
- Проверки безопасности.
```

### Пример использования

```text
Спроектируй ai-weekly-review: вход task_completions за 7 дней, выход summary и 3 рекомендации.
```

## Шаблон 7. Проверка публичной документации

```text
Role:
Ты technical writer и reviewer публичного GitHub-репозитория.

Task:
Проверь <DOC_PATH> перед публикацией.

Context:
- Верхнеуровневые файлы должны выглядеть как продуктовая документация.
- Учебные детали допустимы только в homework/research docs.
- Проверь ссылки, устаревшие пути, противоречия и лишние backstage-формулировки.

Format:
- Найденные проблемы по приоритету.
- Предлагаемые правки.
- Список проверенных ссылок.
- Короткий вывод: готово к публикации или нет.
```

### Пример использования

```text
Проверь README.md и ROADMAP.md перед сдачей ссылки на GitHub.
```
