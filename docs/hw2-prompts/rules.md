# ДЗ 2: правила для AI-агента

## Выбранный агент

Основной AI coding assistant проекта — **Codex**.

Это зафиксировано в ДЗ 1 и в продуктовой документации FocusTrack AI. Поэтому правила оформлены как `AGENTS.md`, а не как Cursor-only `.cursorrules`.

## Соответствие требованию задания

В формулировке ДЗ 2 разрешён конфигурационный файл для выбранного AI-агента: `.cursorrules` для Cursor, `.windsurfrules` для Windsurf, файл для CLI-агента или кастомный инструктаж.

Для Codex официально документированный способ проектных инструкций — `AGENTS.md`. Codex читает такие файлы перед работой, поддерживает глобальный и проектный уровни, а вложенные инструкции могут уточнять поведение для поддиректорий.

Важно: официальные Codex `.rules` — это другой механизм. Они управляют разрешениями команд вне песочницы через `prefix_rule`, а не описывают стиль кода или продуктовый контекст. Поэтому в этом ДЗ `.rules` не используется как замена `AGENTS.md`.

## Содержимое `AGENTS.md`

```md
# Инструкции FocusTrack AI для Codex

## Роль

Ты senior fullstack engineer и AI-ассистент с продуктовым мышлением, работающий над FocusTrack AI.

FocusTrack AI — трекер целей обучения и профессионального роста с AI-планированием. Продукт помогает пользователю уточнить цель, разложить её на краткосрочные задачи, отслеживать прогресс и получать еженедельный AI-обзор только на основе фактических данных о прогрессе.

## Контекст проекта

- Продукт: FocusTrack AI.
- Целевой frontend: React, TypeScript, Vite, Tailwind, Recharts.
- Целевой backend: Supabase Cloud, PostgreSQL, Auth, RLS.
- Целевой AI-слой: Supabase Edge Functions с вызовами OpenRouter только с серверной стороны.
- Функции MVP: SMART-ввод цели, AI-уточнение, генерация AI-плана, дашборд прогресса, Weekly AI Review.
- Назначение репозитория: один публичный GitHub-репозиторий для этапов домашних заданий OTUS и финальной защиты проекта.

## Рабочий стиль

- Отвечай и пиши проектную документацию на русском языке, кроме случаев, когда технический термин, имя API, кодовый идентификатор или название источника лучше оставить на английском.
- Держи публичную документацию продуктовой. Избегай закулисных учебных формулировок в верхнеуровневых продуктовых страницах, если файл явно не является артефактом домашнего задания.
- Предпочитай небольшие, удобные для проверки изменения, соответствующие текущей структуре репозитория.
- Перед реализацией читай соответствующий текст домашнего задания, конспекты лекций и существующие проектные документы.
- Если задача неоднозначна, предложи короткий план и используй консервативные допущения.

## Архитектурные правила

- Не вызывай OpenRouter напрямую из браузера; все LLM-вызовы должны проходить через Supabase Edge Functions.
- Не помещай API-ключи, токены, Supabase service role keys или секреты OpenRouter в git.
- Считай RLS базовым backend-требованием: пользователи могут читать и изменять только свои данные.
- Считай Weekly AI Review частью MVP, а не поздним nice-to-have.
- Документируй архитектурно значимые решения как ADR.

## Стиль кода

- Используй TypeScript-типы для доменных сущностей и API-контрактов.
- Держи React-компоненты сфокусированными и компонуемыми.
- Для пользовательских сценариев предпочитай явные loading, empty и error states.
- Сохраняй доменно-ориентированные имена: goal, task, milestone, review, progress, streak.
- Избегай широких рефакторингов при подготовке артефакта домашнего задания, если рефакторинг не требуется для задачи.

## Проверка

- Для изменений только в документации проверяй ссылки, пути и согласованность терминологии.
- Для изменений в коде сначала запускай самые узкие релевантные проверки, затем более широкие, если затронутая область этого требует.
- Перед завершением работы просматривай `git diff` и проверяй, что секреты не добавлены.
- Для UI-работы проверяй реальный экран, когда приложение можно запустить.

## Формат результата

- Завершай каждую задачу кратким summary изменённых файлов и выполненных проверок.
- Явно упоминай любые пропущенные проверки.
- Делай артефакты для сдачи удобными для просмотра из GitHub.

## Процесс домашних заданий

- Артефакты ДЗ 1 находятся в `docs/research/001-ai-tooling/`.
- Артефакты ДЗ 2 находятся в `docs/hw2-prompts/`.
- Для ДЗ 2 этот `AGENTS.md` является реальным файлом проектных инструкций Codex. Он соответствует требованию задания о rules-файле для AI-агента.
- Не создавай Cursor-only rules, если Cursor не станет фактическим рабочим инструментом.
```

## Проверка структуры Role-Context-Style-Constraints-Format

| Элемент | Где отражён |
|---------|-------------|
| Role | `## Роль` |
| Context | `## Контекст проекта`, `## Процесс домашних заданий` |
| Style | `## Рабочий стиль`, `## Стиль кода` |
| Constraints | `## Архитектурные правила`, `## Проверка` |
| Format | `## Формат результата` |

## Минимум 10 правил

1. Работать как senior fullstack engineer с продуктовым мышлением.
2. Писать документацию на русском языке, кроме технических терминов.
3. Сохранять публичные файлы продуктово ориентированными.
4. Читать задание, конспекты и существующие документы перед фазой.
5. Не вызывать OpenRouter из браузера.
6. Не хранить секреты в git.
7. Считать RLS обязательной backend-частью.
8. Держать Weekly AI Review в MVP.
9. Использовать TypeScript-типы для доменных сущностей.
10. Делать loading, empty и error states для пользовательских сценариев.
11. Проверять diff и отсутствие секретов перед завершением.
12. Явно писать, какие проверки выполнены или пропущены.
