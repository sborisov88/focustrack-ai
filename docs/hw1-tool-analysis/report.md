## 1. Задача и требования

Я выбрал одну идею на весь курс — **FocusTrack AI**, трекер целей обучения с AI-планировщиком.

Подробные требования — в файле `requirements.md`. Коротко: это fullstack-приложение на React и Supabase с AI через OpenRouter и Edge Functions. Проект развивается несколько месяцев, поэтому нужен инструмент для локальной разработки с git, а не одноразовая генерация в облаке.

---

## 2. Сравнительный анализ

Сравнил шесть инструментов: **Bolt.new**, **Lovable**, **Replit Agent** (Cloud) и **Cursor**, **GitHub Copilot**, **Cline** (IDE/CLI).

Полная таблица с баллами, пояснениями по каждой оценке и комментариями — в файле `comparison_table.md`.

Главное наблюдение: cloud-агенты хороши для быстрого MVP, но плохо подходят для backend, миграций, RLS и долгой работы с одним репозиторием. IDE-агенты дают контроль над проектом, и здесь Cursor оказался сильнее остальных.

Отдельно зафиксировал связь с темой занятия про GigaCode/YandexGPT/DeepSeek/Qwen: эти решения важны как LLM-модели или провайдеры для будущего AI-слоя, но не заменяют основной coding-agent workflow для разработки проекта.

---

## 3. Почему выбран Cursor

Подробное обоснование и разбор альтернатив — в файле `tool_selection.md`.

Кратко: FocusTrack AI — monorepo на месяцы вперёд; нужны миграции, RLS, Edge Functions, Cursor Rules (ДЗ 2) и MCP. Cursor набирает 4,7 в таблице против 2,7–3,2 у cloud-агентов и 3,7–3,8 у альтернативных IDE/CLI-инструментов.

## 6. Выводы

1. Для учебного fullstack-проекта на несколько месяцев нужен IDE-агент, не cloud-конструктор.
2. **Cursor** — основной инструмент для FocusTrack AI на весь курс.
3. Cloud-агенты — точечно для UI-моков (ДЗ 3), не как база проекта.
4. Следующий шаг — ДЗ 2: `.cursor/rules` и промпт-шаблоны.
