# Knowledge/RAG experiment

## Цель

Проверить, нужен ли FocusTrack AI слой ответов по документам проекта и учебному контексту.

## Гипотеза

Пользователю полезно задавать вопросы вроде:

- какие критерии сдачи ДЗ 3 ещё не закрыты;
- какие материалы входят в backend-сдачу;
- что показать на защите проекта;
- почему OpenRouter должен вызываться только с сервера.

## MVP-подход

На этом этапе не строится полноценный embedding index. Реализован controlled RAG:

1. frontend или backend передаёт в `rag-answer` список документов;
2. Edge Function отправляет вопрос и документы в OpenRouter;
3. system prompt запрещает отвечать вне переданного контекста;
4. модель возвращает ответ и источники по названиям документов.

## Реализация

```text
supabase/functions/rag-answer/index.ts
```

Таблицы для дальнейшего развития:

- `knowledge_documents`;
- `knowledge_answers`.

## Пример запроса

```json
{
  "question": "Что нужно показать на защите?",
  "documents": [
    {
      "title": "ROADMAP.md",
      "content": "Final: MVP and presentation..."
    }
  ]
}
```

## Критерии оценки

| Критерий            | Ожидание                                       |
| ------------------- | ---------------------------------------------- |
| Ответ по документам | модель ссылается только на переданный контекст |
| Нехватка данных     | модель явно сообщает, что данных недостаточно  |
| Стоимость           | один chat completion без embedding pipeline    |
| Риск                | ниже, чем у полного RAG, но хуже recall        |

## Решение по MVP

RAG остаётся экспериментом и демонстрационным плюсом. В core MVP входят цели, задачи, прогресс и weekly review. Полноценный RAG на pgvector стоит добавлять после защиты, если пользователю действительно нужно искать по большому архиву документов.
